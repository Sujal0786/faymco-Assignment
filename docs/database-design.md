## 2. Table Specifications & Fields

### Table: `users`
Tracks the registration of affiliate publishers.
- `id` (UUID): Primary Key.
- `username` (VARCHAR): Unique username.
- `email` (VARCHAR): Unique email.

### Table: `brands`
Tracks brands whose products are sold by affiliate users.
- `id` (UUID): Primary Key.
- `name` (VARCHAR): Unique brand name.

### Table: `sales`
Tracks individual affiliate conversions, their status, earnings, and advance payout status.
- `id` (UUID): Primary Key.
- `user_id` (UUID): Foreign Key -> `users(id)`.
- `brand_id` (UUID): Foreign Key -> `brands(id)`.
- `external_reference` (VARCHAR): Unique reference ID generated by external sales tracking systems.
- `status` (VARCHAR): Enforced checklist constraint: `pending`, `approved`, `rejected`.
- `earning_amount` (BIGINT): Earning commission in integer paise. Must be `>= 0`.
- `advance_paid_amount` (BIGINT): Amount already transferred as advance. Must be `>= 0`.
- `reconciled_at` (TIMESTAMP): Time when status was updated to approved/rejected.
- `version` (INT): Concurrency control version.

### Table: `wallets`
Represents the user's withdrawable ledger balance.
- `id` (UUID): Primary Key.
- `user_id` (UUID): Foreign Key -> `users(id)`, UNIQUE constraint ensures 1-to-1 mapping.
- `withdrawable_balance` (BIGINT): Current withdrawable funds (can be negative due to rejected adjustments).
- `reserved_balance` (BIGINT): Funds locked for processing withdrawals. Must be `>= 0`.
- `version` (INT): Concurrency control version.

### Table: `payouts`
Tracks external payouts sent to users via payment provider.
- `id` (UUID): Primary Key.
- `user_id` (UUID): Foreign Key -> `users(id)`.
- `type` (VARCHAR): `advance`, `withdrawal`, `final`.
- `amount` (BIGINT): Payout value in paise.
- `status` (VARCHAR): `created`, `processing`, `succeeded`, `failed`, `cancelled`, `rejected`.
- `provider_reference` (VARCHAR): Transaction reference returned by payout provider.
- `idempotency_key` (VARCHAR): Unique key to avoid double-processing payouts.

### Table: `payout_allocations`
Maps individual payouts to their contributing sales, allowing tracking of advances.
- `id` (UUID): Primary Key.
- `payout_id` (UUID): Foreign Key -> `payouts(id)`.
- `sale_id` (UUID): Foreign Key -> `sales(id)`.
- `allocation_type` (VARCHAR): `advance`, `final`.
- `amount` (BIGINT): Allocated paise amount.
- `status` (VARCHAR): `created`, `processing`, `succeeded`, `failed`, `cancelled`, `rejected`.

### Table: `wallet_ledger_entries`
Immutable double-entry book-keeping ledger auditing all wallet balance movements.
- `id` (UUID): Primary Key.
- `user_id` (UUID): Foreign Key -> `users(id)`.
- `sale_id` (UUID): Foreign Key -> `sales(id)` (Optional).
- `payout_id` (UUID): Foreign Key -> `payouts(id)` (Optional).
- `entry_type` (VARCHAR): `ADVANCE_PAYOUT`, `APPROVED_SALE_REMAINDER`, `REJECTED_SALE_ADJUSTMENT`, `WITHDRAWAL_RESERVED`, `WITHDRAWAL_COMPLETED`, `WITHDRAWAL_RELEASED`, `FAILED_PAYOUT_REFUND`.
- `amount` (BIGINT): Balance adjustment in paise (signed).
- `balance_after` (BIGINT): Account balance after transaction (signed).
- `idempotency_key` (VARCHAR): Unique constraint to block duplicate postings.

### Table: `withdrawal_requests`
Tracks manual cash withdrawals requested by users.
- `id` (UUID): Primary Key.
- `user_id` (UUID): Foreign Key -> `users(id)`.
- `amount` (BIGINT): Withdrawal amount. Must be `> 0`.
- `status` (VARCHAR): `created`, `processing`, `succeeded`, `failed`, `cancelled`, `rejected`.
- `retry_of_withdrawal_id` (UUID): Foreign Key -> `withdrawal_requests(id)` (Optional) linking immediate retries.
- `idempotency_key` (VARCHAR): Unique idempotency key.

### Table: `processed_webhook_events`
Enforces idempotency on payment gateway webhooks.
- `id` (UUID): Primary key.
- `provider_event_id` (VARCHAR): Unique event ID from provider.
- `event_type` (VARCHAR): E.g., `payout.failed`.
- `payload` (JSONB): Entire webhook payload stored for audit.

## 3. Database Constraints & Optimizations

### Check Constraints
- `sales_status_check`: `status IN ('pending', 'approved', 'rejected')`
- `sales_earning_amount_check`: `earning_amount >= 0`
- `sales_advance_paid_amount_check`: `advance_paid_amount >= 0`
- `wallets_reserved_balance_check`: `reserved_balance >= 0`
- `payouts_type_check`: `type IN ('advance', 'withdrawal', 'final')`
- `payouts_status_check`: `status IN ('created', 'processing', 'succeeded', 'failed', 'cancelled', 'rejected')`
- `payouts_amount_check`: `amount >= 0`
- `withdrawal_requests_amount_check`: `amount > 0`

### Partial Unique Index
Enforces that a sale can only ever receive **one successful advance payout**:
```sql
CREATE UNIQUE INDEX unique_successful_advance_allocation 
ON payout_allocations (sale_id) 
WHERE (allocation_type = 'advance' AND status = 'succeeded');
```

### Performance Indexes
- `idx_sales_user_status` on `sales (user_id, status)` - Speeds up sale queries.
- `idx_sales_status_advance` on `sales (status, advance_paid_amount)` - Speeds up advance job queries.
- `idx_payouts_user_status` on `payouts (user_id, status)` - Optimizes payout tracking queries.
- `idx_withdrawal_user_req` on `withdrawal_requests (user_id, requested_at)` - Speeds up the 24-hour limit checks.
- `idx_ledger_user` on `wallet_ledger_entries (user_id)` - Speeds up ledger queries.
